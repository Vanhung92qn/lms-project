# ADR-001 — Tech Stack Selection

- **Status:** Accepted
- **Date:** 2026-04-19
- **Deciders:** Product owner, Tech Lead (AI agent on their behalf)
- **Supersedes:** —

## Context

We are building an AI-LMS for `khohoc.online` from scratch. Constraints:

- Pilot target: <500 students, <50 concurrent.
- Budget: 1 VPS (16 GB RAM / 4–8 vCPU / no GPU).
- Team: 1 product owner + 1 autonomous AI engineer.
- Domain complexity is moderate (DDD-friendly), traffic profile is
  modest, but the feature surface is large: CMS, auth, code sandbox,
  AI tutor, billing, analytics.

We need a stack that is **boring, hire-able, fast to build on, and
friendly to a single-node deployment**, without painting us into a
corner when we later need to scale a specific hot spot.

## Decision

Adopt the following stack:

| Layer | Choice |
|-------|--------|
| Monorepo | pnpm workspaces + Turborepo |
| Web | Next.js 14 (App Router) + React 18 + TypeScript |
| UI | TailwindCSS + shadcn/ui + Radix primitives |
| Code editor widget | Monaco (`@monaco-editor/react`) |
| i18n | `next-intl` |
| Core API | NestJS 10 + TypeScript, modular monolith |
| API contract | OpenAPI 3.1 via `@nestjs/swagger`, client generated by `openapi-typescript` |
| Auth | JWT RS256 + refresh rotation + Casbin for RBAC |
| Primary DB | PostgreSQL 16 + `pgvector` extension |
| ORM | Prisma |
| Document DB | MongoDB 7 |
| Cache / queue | Redis 7 + BullMQ |
| Code sandbox | Custom Docker runners orchestrated from Python (`docker-py`) |
| Local AI | Ollama + Llama 3 8B Instruct Q4\_K\_M (GGUF) |
| Cloud AI | Google Gemini (`gemini-1.5-flash`) |
| Python services | FastAPI + Uvicorn + Pydantic v2 |
| Reverse proxy / API gateway | Traefik v3 |
| Edge | Cloudflare Free (DNS, WAF, Origin CA, proxied) |
| Observability | Prometheus + Grafana + Loki + Sentry (free tier) |
| CI/CD | GitHub Actions + GHCR |
| Deploy | Docker Compose (single-node) |
| Payments | VNPay + MoMo (Stripe post-MVP) |

**Explicitly rejected from the MVP** (to be revisited via their own
ADRs when / if demand materializes):

- **Neo4j** — Postgres graph tables + recursive CTEs cover the v1
  Knowledge Graph. See ADR-003.
- **Kubernetes / K3s / service mesh** — overkill at <50 concurrent
  users. Docker Compose is the right tool. See ADR-002.
- **Kafka / RabbitMQ** — BullMQ on Redis handles two queues at this
  scale without the operational tax.
- **Elasticsearch** — Postgres full-text search is sufficient.
- **Judge0** — we want fine-grained control over sandbox security
  (seccomp, userns, image minimalism). See ADR-004.

## Consequences

### Positive

- Every piece is either first-party supported (Postgres, MongoDB,
  Redis, Docker, NestJS, Next.js) or has a huge community (Prisma,
  BullMQ, Ollama). Operational docs are abundant.
- Most of the stack is free / OSS; the only recurring cost is the VPS.
- The monolith-first approach keeps DDD boundaries enforceable via
  NestJS modules, and extraction to a service is a packaging step
  rather than a redesign.
- The TypeScript-heavy stack means shared types between web and API
  are cheap — one source of truth (OpenAPI) flows into both.

### Negative

- 16 GB RAM is tight with Ollama loaded (≈ 6 GB). We will live close
  to the ceiling; observability is non-optional.
- Llama 3 8B on CPU is slow (10–30 s per response). We mitigate with
  SSE streaming + Gemini fallback (see ADR-005 / ai-integration.md).
- Python services add a second language runtime to operate. We accept
  this because Python wins clearly for Docker orchestration, AI, and
  numerics — the alternative (JS everywhere) would cost us more in
  workarounds than polyglot costs us in container images.
- Prisma's migration story is good but not great under heavy schema
  churn — we'll lean on small, frequent migrations.

## Alternatives considered

- **Full microservices (Auth, Course, Progress, …) from day 1.**
  Rejected: infra + mental overhead crushes a solo team for no
  scaling benefit at pilot traffic.
- **Rails/Django monolith.** Faster to scaffold, but we give up
  shared-types with the web tier and lose the Node ecosystem for the
  Monaco-integration work.
- **SQL Server.** Familiar to the product owner, but licensing on
  Linux is a friction point and Postgres' extension ecosystem
  (pgvector, PostGIS-if-ever, `pg_stat_statements`) is unmatched.
- **Gemini-only AI.** Simpler, zero hosting cost, but leaks student
  code to a third party and removes the ability to learn on top of
  our data later. Hybrid is worth the ops cost.
